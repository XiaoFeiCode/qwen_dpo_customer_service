# Interview Notes

## 你要怎么介绍这个项目？

这个项目面向电商客服多轮对话，目标是解决客服回复机械、拒答边界不稳定、回复偏好不一致的问题。我把项目拆成四层：第一层是原始客服数据清洗，构造 SFT 样本和 DPO 偏好对；第二层是基于 Qwen 的 LoRA/QLoRA SFT，让模型先学会电商客服话术和基础售后流程；第三层是用 chosen/rejected 偏好对做 DPO，让模型更偏向有用、礼貌、合规的回复；第四层是离线评估和 API 服务，从无效回复率、拒答准确率、偏好对胜率等指标做 badcase 迭代。

## 为什么不是直接 SFT？

SFT 能让模型学会客服场景的基本回复格式，但它很难显式表达“哪种回复更好”。比如两个回复都能回答问题，一个可能更礼貌、更合规、更可执行，另一个可能很机械或者越权承诺。DPO 用 chosen/rejected 偏好对直接优化这种偏好差异，比只做 SFT 更适合客服回复质量优化。

## chosen/rejected 怎么构造？

我按客服场景构造偏好对：chosen 要满足礼貌、边界清晰、动作可执行、必要字段明确；rejected 通常包含机械敷衍、过度承诺、隐私越权、售后规则不清晰等问题。比如涉及手机号、身份证、验证码时，chosen 会明确拒绝并引导官方流程，rejected 则是违规帮用户查询隐私。

## 为什么要只对 assistant 回复计算 loss？

多轮对话训练里，system 和 user 部分是条件，不是模型应该学习生成的目标。如果把用户问题也作为 loss，会让模型学习复述用户输入，浪费训练信号。只对 assistant 回复计算 loss，可以让梯度集中在客服回答质量上。

## DPO 的 beta/KL 起什么作用？

beta 控制策略模型相对参考模型的偏移强度。beta 太小，模型可能过度追逐偏好数据导致风格漂移；beta 太大，模型更新保守，偏好优化不明显。面试时可以说我会从 0.05、0.1、0.2 做小规模对比，观察拒答准确率、无效回复率和人工偏好胜率。

## 评估怎么做？

我设计了四类离线指标：

- 人工偏好胜率：比较 SFT baseline 和 DPO 模型回复，人工判断哪一个更好。
- 无效回复率：统计“自己看、我不知道、找客服吧”等短且无帮助的回复。
- 拒答准确率：该拒绝时是否拒绝，不该拒绝时是否过度拒绝。
- 多轮一致性：多轮场景下是否保留订单、售后诉求、商品状态等上下文。

当前仓库提供规则评估器作为 smoke test，真实训练时应补人工评测表和 badcase 标注。

## 面试官可能追问

1. DPO 和 PPO/RLHF 的区别是什么？
   DPO 不需要训练 reward model，也不需要在线采样 PPO 更新，而是直接用偏好对优化策略模型，工程链路更简单，适合中小规模偏好对齐。

2. 你的偏好数据量不大怎么办？
   先用小规模高质量偏好对验证流程，再按 badcase 扩充数据。客服场景更看重边界和流程，少量高质量样本比大量噪声样本更有价值。

3. 怎么避免模型乱承诺退款？
   数据层面加入售后规则和拒答边界样本；训练层面用 DPO 惩罚越权承诺；推理层面可以加规则校验或检索平台规则。

4. QLoRA 和 LoRA 的区别？
   LoRA 是只训练低秩适配器，QLoRA 会把底座模型量化后再训练 adapter，显存成本更低，适合单卡资源不足时微调大模型。

5. 这个项目最大的工程价值是什么？
   不是单纯调用模型，而是把数据构建、偏好对齐、评估指标和服务接口串成闭环。面试时重点讲“如何发现 badcase、如何转成偏好对、如何通过指标验证改善”。
